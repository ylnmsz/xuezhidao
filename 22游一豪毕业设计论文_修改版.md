# 基于 Web 的题库练习与错题本系统

物联网工程专业学生 游一豪

指导教师 刘振

**摘要：** 在教育信息化持续推进的过程中，传统题库管理与试题录入方式效率偏低，对于包含文字、图片与数学公式的理科资料，数字化处理难度较大，教师需要花费大量时间手动整理试卷内容。为解决这类实际教学问题，本文设计并实现了一套基于 Web 的题库练习与错题本系统——学之道。

本系统采用前后端分离的开发模式，前端使用 Vue 3 与 Vite 构建页面，借助 KaTeX 完成数学公式的渲染工作，采用 Tailwind CSS 实现界面样式；后端以 Node.js 和 Express 为框架，使用 PostgreSQL 完成数据存储工作。系统结合 MinerU 文档解析工具与大语言模型，通过切片处理与容错解析方案，实现 Word 试卷的自动上传、内容拆分与题库生成。同时系统集成作业发布、在线练习、错题整理、AI 辅助答疑等功能，为教师教学与学生自主学习提供便捷的数字化支持。

经过功能与性能测试，本系统可稳定处理图文混排的试卷文档，解析速度与准确率满足日常使用需求，能够有效减少教师的试卷整理工作量，提升学生的习题练习效率，具备一定的实际使用与推广价值。

**关键词：** 教辅平台；大语言模型；MinerU；Vue 3

# 1 绪论

## 1.1 研究背景

当前，教育数字化已成为中小学教学的重要发展方向。国家中小学智慧教育平台自上线以来，累计注册用户突破1.6亿，浏览量超过608亿次，基于该平台的全国30,605名中小学生调研显示，75.4%的学生将其用于课程教学，线上教育资源已被广泛应用于日常教学[3]。但在实际使用中，多数教学平台仍存在明显短板：教师整理理科试卷时，需要手动输入题目、公式与图表，耗时较长；学生完成练习后，错题整理依赖手动记录，复习效率较低；传统平台缺少自动化解析与个性化学习辅助功能。

在教育数字化背景下，智能教辅工具已成为提升教学效率的重要手段。在智能试题处理方面，Ruijun等人提出了一种层次化答案感知与上下文感知的试题生成网络（HACAN），通过融合答案信息与上下文语义特征，实现了从文本段落中自动生成高质量阅读理解问题，为智能教辅系统中的试题自动构建提供了新的技术路径[6]。在个性化学习服务方面，Chen等人提出了一种基于在线学习风格模型的自适应推荐方法（AROLS），通过聚类学习者的在线学习行为特征并融合协同过滤与关联规则挖掘，实现了教育资源的个性化精准推荐，为智能教辅系统中的个性化学习支持提供了技术参考[18]。同时，大语言模型在学科知识问答、试题生成、编程学习等教育场景中的应用日趋深入，正推动人机对话式学习新形态的形成[4]。将智能化技术应用于题库与错题本系统已成为可行方案。基于此，本文开发一套轻量化、易使用的 Web 题库练习与错题本系统，通过自动化工具简化试卷处理流程，为师生提供一体化的习题练习与管理服务。

## 1.2 研究目的与意义

### 1.2.1 研究目的

本课题要做的事情很明确：给中小学师生做一套能真正用起来的教辅工具，重点是解决两个老问题——试卷处理太慢、个性化学习支持不够。现在的情况是，老师拿到一份理科试卷，里面又是公式又是图，想把它变成电子版的题库就得逐道题手动录入，光这一项就能耗掉好几个小时。所以第一个目标就是打通自动解析这条路，结合文档版面分析工具和大模型，让老师上传一份 DOCX 或 PDF 试卷之后，系统自己把题目拆好、公式认好、图片对应好，直接入库，不用人再折腾。第二个目标是把整个教学流程串起来——题库整理好了，老师能方便地从中选题组卷发给学生，学生在线上答完题客观题能自动出分，做错的题自动收进错题本方便以后复习，这样一来教学的前前后后在一个平台上就能走完。第三个目标跟大模型的使用有关。大模型虽然能理解文本内容，但处理几页纸的长文档时经常要么响应慢要么输出格式乱七八糟，直接拿来用不太靠谱。所以还得想办法把长文档切成合适的小块，同时写一套能兜底的 JSON 解析逻辑，保证大模型偶尔出点岔子时系统不至于崩溃。说到底，就是想做一款老师愿意用、学生用得惯的轻量级工具，让技术真正帮教学节约时间，而不是添麻烦。

### 1.2.2 研究意义

实际应用层面，本系统可直接用于日常教学练习，教师可快速上传试卷生成作业，学生可随时进行习题练习与错题巩固，提升教与学的效率。

技术研究层面，本系统将文档解析、大模型应用与 Web 开发结合，为同类轻量化教学工具的开发提供了可参考的实现方案。

## 1.3 国内外研究现状

### 1.3.1 国外研究现状

国外教育类平台发展较早，Quizlet、Google Classroom 等工具可实现题库管理、作业分发等基础功能，在界面交互与功能模块化方面较为成熟。但这类平台针对中文试卷、数理公式的适配性较差，且本地化部署难度较高，难以直接用于国内教学场景。此外，在学习行为分析与预测方面，Xie对MOOC学习者的观看时长数据进行了生存分析，发现学习者的辍学风险函数呈现浴盆曲线特征，并构建了随机微分方程模型来刻画不同类别学习者的学习时长增长模式，为教育平台中学习者行为的理解与学业预警提供了方法参考[17]。

国外的研究也不光停留在平台层面。Khan Academy 这些年积累了海量的学习行为数据，靠的就是知识点掌握度追踪和个性化推荐这套机制，这些数据反过来又推动了自适应学习算法的改进。智能导师系统这边，早期基于规则的方案慢慢被数据驱动的方法取代了，贝叶斯知识追踪和深度知识追踪这类模型开始流行，能用数据动态判断学生知识点掌握到什么程度，再针对性推题。Duolingo 做得比较成功的一点是把游戏化机制嵌到了学习流程里——积分、等级、排行榜这些手段确实能让人坚持学下去，这种思路放到教育类Web应用里挺有参考价值。文档分析方面，像 MathPix 这类工具已经能识别带公式的图片并转成 LaTeX，公式识别这块技术确实比较成熟了，不过对中文试卷的版式还原还是差点意思。

### 1.3.2 国内研究现状

国内现有题库类平台资源丰富，但自动化水平有限，试题录入仍以人工为主。部分平台支持图片识别，但对复杂公式、试卷版式的还原效果较差。在技术研究方面，陈欣等人利用大语言模型结合检索增强与提示工程，实现了试题自动生成，合格率达 86.47%[1]。在自动阅卷技术方面，王睿等人基于神经网络OCR技术研究了自动阅卷系统中的文字识别方法，为在线教育平台的客观题自动批改提供了技术支撑[5]。李淑玲等人基于 Node.js 技术设计并实现了在线测试系统，支持自动判分与成绩统计[7]。王玲研究了高并发处理模式，将 Node.js 与 Redis 缓存结合，支撑了近 70 万考生的在线考试[14]。此外，基于 MEVN 技术栈（MongoDB、Express、Vue、Node.js）的考试系统也被应用于计算机等级考试中[2]。在智能化试题组织方面，Yu等人提出了一种结合知识追踪与强化学习的试题自动组卷框架，通过分析学习者的知识掌握状态动态调整试题组合策略，为智能教辅平台的个性化练习与精准测评提供了技术支持[11]。在智能化数据处理方面，Hu等人提出了一种基于密集作业集优先（DJSF）的启发式作业调度方法，通过最大化单位时间内完成的作业数量，有效降低了平均作业完成时间并提升了大数据分析场景下的系统吞吐量，为教育平台中高并发数据处理与资源调度提供了技术参考[16]。人工智能技术也被应用于物理学科的错题库建设中，通过智能分类与管理提升错题复习效率[8]。

再看国内，学而思和猿辅导这类商业平台题库确实大、用户数据也多，但系统基本都是闭源的，想在上面做定制或者二次开发很麻烦，普通学校很难拿来直接用。开源的考试系统倒是有不少，Spring Boot 和 Node.js 的都有人做过，不过大多还是管考试流程那一套，真正能把复杂试卷自动解析成结构化的少之又少。错题本这块，有些研究开始用知识图谱来做错题分析了——先搭一个学科的知识点网络，然后把学生的错题和薄弱知识点挂上钩，这样推复习内容就更准一些。公式渲染方面，Web 上已经有 KaTeX 和 MathJax 这样的成熟方案，理科题目里的公式展示基本没什么问题。架构上，Vue.js 加 Node.js 这种前后端分离的组合现在做教育系统的人用得越来越多，组件化开发维护方便，并发处理也不差。

### 1.3.3 研究现状总结

总的来看，现在教育平台和题库系统这块，国内外各有各的问题。国外平台像 Quizlet、Google Classroom 交互做得不错、功能模块也清楚，但到了中文环境和数理公式这块就水土不服。国内平台题库资源其实不少，智能化这些年也在往前走，但大部分还是靠人工录题，真正能自动解析的没几家。技术路线上，大语言模型和文档版面分析这两个东西怎么结合起来用，目前还在探索阶段，能把它们真正落地到试卷自动解析这个场景的就更少了。错题本和学情分析的研究，多数还停留在记录和展示数据这一步，跟 AI 答疑或者个性化推荐结合得不够紧。

针对这些问题，本文选了一条比较务实的技术路线——用轻量化的技术栈，把 MinerU 的版面分析能力、大模型的文本抽取能力和自己写的高容错解析算法串在一起，做成一条从上传试卷到题库入库的自动化流水线。同时把题库管理、作业分发、在线作答、错题本和 AI 答疑这些功能也打通，形成一个完整的教辅闭环，尽量补齐现有工具在智能化和工程落地之间的缺口。

# 2 系统核心技术与理论基础

## 2.1 前端开发技术

### 2.1.1 Vue 3 与 Composition API

本平台使用 Vue 3 作为前端核心框架，基于 Composition API 实现组件化开发与状态管理。马利军研究了Web前端架构模式的演化历程，分析了MVVM模式在主流前端框架中的应用优势[9]。

### 2.1.2 Vite 构建工具

Vite 具有启动快、热更新高效的特点，支持代码按需加载，优化页面加载速度，适合构建轻量化 Web 应用。

### 2.1.3 状态管理

本平台未引入 Pinia 或 Vuex 等全局状态管理库，而是采用 Vue 3 的 Composition API（`ref`、`computed`）结合 `localStorage` 实现用户状态持久化。对于组件间通信，采用 props 与 emit 机制，保持架构轻量简洁。

### 2.1.4 KaTeX 公式渲染引擎

KaTeX 可快速渲染数学公式，支持分式、积分等常见数理格式，解决理科试题中公式显示不规范的问题。

### 2.1.5 Tailwind CSS

本平台使用 Tailwind CSS 作为界面样式方案。Tailwind CSS 是一个原子化（utility-first）的 CSS 框架，通过预定义的语义化类名快速构建界面，开发效率高、样式一致性强。配合 Material Symbols 图标字体，实现了美观、统一的视觉风格。

## 2.2 后端开发技术

### 2.2.1 Node.js

Node.js 是基于 Chrome V8 引擎的 JavaScript 运行环境，采用事件驱动、非阻塞 I/O 模型，适用于处理文件上传、网络请求、数据计算等 I/O 密集型场景。本平台使用 Node.js 作为后端运行环境，统一前后端编程语言，降低开发成本。

### 2.2.2 Express 框架

Express 是 Node.js 生态中最成熟的 Web 开发框架，轻量级、易扩展，提供路由、中间件、请求处理等核心功能。本平台基于 Express 搭建后端服务，开发 RESTful API 接口。

### 2.2.3 JWT 身份认证

JWT（JSON Web Token）用于用户登录状态验证，无需存储会话信息，可实现教师与学生的权限区分，保障系统访问安全。雷云祥等人基于 JWT 设计了"集中认证、分散鉴权"的分布式权限控制系统[10]。闵晓霜等人将 JWT 与 Spring Security 结合实现了统一身份认证系统[12]。本平台后端使用 jsonwebtoken 中间件完成 Token 签发与验证，配合路由守卫实现角色权限控制。

### 2.2.4 Multer 文件处理

Multer 是 Express 框架中常用的文件上传中间件，可对上传的 Word、PDF 试卷进行格式校验与存储，为文档解析提供数据基础。

## 2.3 文档版面分析技术

### 2.3.1 MinerU 引擎原理

MinerU 是一款开源的文档版面分析工具，可对 PDF、DOCX 等文档进行视觉分析，分离文字、图片与公式，保留原有版式并输出 Markdown 格式的结构化内容。相比传统 OCR 技术，MinerU 在试卷类文档的解析上具有版式保留完整、图文对位精准的优势。

### 2.3.2 文档解析技术优势

传统文档处理技术仅能提取纯文本，破坏版式与公式；MinerU 可完整保留试卷原有结构，实现图文精准对位，为后续大模型结构化抽取提供高质量输入数据，是本平台实现试卷智能解析的核心技术支撑。

## 2.4 大语言模型应用技术

### 2.4.1 大语言模型选型

系统选用兼容 OpenAI API 协议的开源大语言模型，具备文本理解与信息抽取能力，可从解析后的文档中提取题干、选项、答案等内容。

### 2.4.2 Prompt 工程设计

通过设定清晰的提取指令和输出格式约束，规范大模型输出结构，减少内容错乱，提升试题抽取的准确性。

### 2.4.3 大模型应用场景

1. 试卷试题结构化抽取；
2. AI 助教答疑，提供解题思路；
3. 试题难度与知识点自动标注；
4. 错题解析与个性化学习建议生成。

## 2.5 数据库技术

### 2.5.1 PostgreSQL 数据库

PostgreSQL 是一款功能强大的开源关系型数据库，稳定性强，支持 JSON/JSONB 等复杂数据类型，适合存储试题、用户、作业、错题等多样化数据。在数据库查询优化方面，Wang等人提出了一种在有限主存环境下进行空间数据高效查询的方法，通过有界内存中的top/bottom-k分数估算算法，实现了大规模空间数据的快速检索，为教育平台中试题数据的高效查询提供了技术参考[15]。在数据库存储优化方面，Liu等人提出了基于多时钟快照隔离（MCSI）的并发控制机制，通过向量时钟与多版本存储在非易失性内存数据库中实现了高并发事务处理，在YCSB测试中获得了101%–195%的吞吐量提升，为教育场景下大规模数据的高效存储与访问提供了技术参考[13]。

### 2.5.2 数据库设计原则

数据库设计这块，主要考虑了四个方面的东西。首先是数据一致性，用户、试题、作业这些表之间的关联不能乱，尤其是外键引用和级联操作要定清楚，不然数据对不上后面查起来很麻烦。然后是扩展性，开发的时候很难一下子把所有字段都想全，所以表结构里适当预留了一些余地，将来要加学科、题型或者新的功能模块时不用大改表结构。查询效率也是必须考虑的点，试题表和用户表的数据量上来之后，如果没有索引检索会越来越慢，所以在常用的查询条件上都建了索引来保证响应速度。最后是数据安全，用户密码不能明文存，试题和成绩数据也得有权限控制，不同角色能看的东西不一样，这些在设计阶段就定好了规则。

# 3 系统需求分析

## 3.1 需求调研与分析

### 3.1.1 调研对象与调研方法

#### 调研对象

本次调研面向中小学教师与学生，旨在了解日常教学中题库使用、作业布置、习题练习与错题整理的实际情况。调研共发放问卷 200 份，回收有效问卷 178 份，其中教师问卷 82 份、学生问卷 96 份。教师受访者覆盖语文、数学、英语、物理、化学等主要学科，教龄分布涵盖新入职教师（1-3年）至资深教师（10年以上）；学生受访者涵盖初一至高三六个年级，确保样本具有代表性。

#### 调研方法

调研采用问卷调查与个别访谈相结合的方式。问卷调查主要收集量化数据，包括教师日常试卷处理耗时、现有工具使用满意度、功能需求优先级等指标；个别访谈则选取了 5 名教师和 8 名学生进行半结构化访谈，深入了解其在教学与学习过程中遇到的具体困难与改进期望。

### 3.1.2 调研结果

调研数据从多个维度揭示了当前教辅工作中的实际状况，主要结果如下：

**教师端调研数据：**

- 95% 的教师认为试卷录入耗时过长，其中超过六成教师表示每周需要花费 3 小时以上手动录入或整理试题。
- 88% 的教师希望实现试卷自动解析功能，尤其是数理学科教师对公式识别的需求最为迫切。
- 82% 的教师需要自动化作业分发与学情分析功能，认为批改作业和统计成绩占用了大量教学准备时间。
- 76% 的教师反映现有题库系统操作复杂，希望拥有更简洁易用的管理界面。

**学生端调研数据：**

- 90% 的学生希望拥有智能错题本，避免手动抄写错题的低效方式。
- 76% 的学生需要实时答疑工具，在做题遇到困难时能够及时获得解题思路引导。
- 85% 的学生喜欢简洁易用的在线答题界面，认为界面美观度会影响练习积极性。
- 68% 的学生表示希望看到自己的学习进步轨迹和排名数据，以此激励自己持续学习。

**核心痛点归纳：**

1. 试卷处理效率低：从纸质或电子版试卷到结构化题库，中间环节多、人工操作量大。
2. 公式处理难：现有工具对数学公式、化学方程式的输入与展示支持不足。
3. 错题整理麻烦：学生需要手动摘抄错题，缺乏系统化的自动收录与分类。
4. 答疑不及时：遇到难题时缺少即时有效的答疑渠道，影响学习连贯性。
5. 学情数据缺失：教师难以获取班级整体的知识点掌握情况，教学调整缺乏数据支撑。

### 3.1.3 需求提炼

基于上述调研结果，将师生核心需求映射为系统功能模块，提炼出以下关键需求：

1. **试卷智能解析需求**：系统应能够自动解析上传的 DOCX/PDF 试卷，完成图文分离、公式识别与结构化数据抽取，将传统数小时的人工录入缩短至分钟级。
2. **题库管理需求**：支持试题的编辑、检索、分类与批量管理，为教师提供完整的题库维护功能。
3. **作业分发与批改需求**：支持教师在线组卷、发布作业，学生在线作答后客观题自动批改，主观题支持教师手动评阅。
4. **在线作答需求**：提供沉浸式答题环境，支持公式、图片的正常显示与流畅交互。
5. **智能错题本需求**：答题错误后自动收录错题，支持按学科、知识点分类筛选与错题重做。
6. **学情分析需求**：以可视化图表展示作业完成率、正确率、薄弱知识点等数据，辅助教师调整教学策略。
7. **权限管理需求**：实现教师端与学生端的角色权限分离，确保数据访问安全。

### 3.2.3 核心功能需求详述

针对系统核心功能进行更详细的需求描述，作为后续设计与实现的重要依据。

**1. 智能文档解析**

智能文档解析是本系统的核心功能，旨在将教师上传的试卷文档自动转换为结构化题库数据，具体需求如下：

- **文件输入**：支持 DOCX、PDF 两种常见试卷格式的上传，单个文件大小不超过 50MB。
- **图文分离**：调用 MinerU 引擎对文档进行版面分析，将文字、图片、公式从文档中分离提取。
- **图片持久化**：将解析出的图片资源持久化存储至服务器，并生成可访问的资源链接替换文档中的图片占位符。
- **文本切片**：对长文档按试题边界进行智能切割，每一切片包含一道完整试题。
- **结构化抽取**：调用大语言模型对每个切片进行结构化抽取，输出包含题型、题干、选项、答案、解析的 JSON 数据。
- **容错处理**：当大模型输出格式异常时，执行多级容错解析策略，包括正则修复、反斜杠转义修复、JSON 字段补全等。
- **入库存储**：解析完成后的试题自动写入数据库，并关联至对应的题库目录。

**2. 公式渲染**

- 支持 LaTeX 格式的数学公式渲染，涵盖分式、积分、根式、求和、方程组、矩阵等常见数理格式。
- 支持化学方程式与符号的渲染显示。
- 在题干、选项、答案解析等多处位置统一调用 KaTeX 渲染引擎，确保公式显示清晰、无错位与乱码。

**3. 权限控制**

- 基于 JWT 实现无状态身份认证，Token 过期后自动要求重新登录。
- 前端路由守卫配合后端中间件，实现教师端、学生端、管理员端的视图与接口权限隔离。
- 未登录用户仅可访问登录注册页面，无法访问系统核心功能页面与数据接口。

## 3.3 非功能性需求分析

非功能性需求是系统能否在实际教学场景中长期稳定运行的关键保障，下面从性能、安全、易用性、可扩展性与可靠性五个维度进行详细分析。

### 3.3.1 性能需求

性能指标直接影响用户体验与系统的实用价值，具体量化指标如下：

1. **页面响应速度**：系统主要页面的首次加载时间不超过 2 秒，接口请求平均响应时间不超过 1 秒。对于包含大量公式和图片的试题展示页面，加载时间不超过 3 秒。
2. **试卷解析速度**：一份包含 20-30 道试题的标准理科试卷，从上传到入库完成的总耗时不超过 60 秒。解析时间应随试卷页数线性增长，而非指数增长。
3. **并发能力**：系统需支持至少 100 名用户同时在线使用，在并发场景下页面操作仍保持流畅，无明显卡顿延迟。
4. **解析准确率**：试题结构化抽取的准确率不低于 95%，公式与图片的识别与还原准确率不低于 99%。对于标准版式的试卷文档，准确率应达到更高水平。
5. **AI 答疑响应**：AI 答疑功能的响应时间不超过 5 秒，保证对话的连贯性与实时性。

### 3.3.2 安全性需求

考虑到系统涉及用户个人信息、试题数据与学业成绩等敏感数据，安全性设计至关重要：

1. **数据安全**：用户密码采用 bcrypt 算法进行哈希加密存储，不可逆向还原。试题数据、成绩数据等核心业务数据在数据库层面定期备份，防止因硬件故障导致数据丢失。
2. **身份安全**：采用 JWT（JSON Web Token）无状态认证方案，Token 设置合理有效期，过期后自动失效。教师与学生之间的数据访问权限严格隔离，防止越权操作。
3. **接口安全**：后端 API 接口在关键业务操作（如删除试题、修改成绩）上进行二次权限校验，防止通过直接调用接口进行越权操作。
4. **文件安全**：对用户上传的文件进行严格的格式校验与大小限制，仅允许 DOCX、PDF 等白名单格式，避免可执行文件或恶意脚本的上传。
5. **传输安全**：系统部署时配置 HTTPS 加密传输，防止数据在传输过程中被截获或篡改。

### 3.3.3 易用性需求

面向中小学师生群体，系统的易用性直接决定了用户采纳率与实际使用效果：

1. **操作简洁**：界面布局清晰，功能入口直观，核心操作（上传试卷、开始练习、查看错题）不超过 3 步即可完成。系统遵循极简设计原则，避免信息过载。
2. **交互反馈**：用户操作后系统提供及时的状态反馈，如上传进度条、解析状态提示、提交成功通知等。错误发生时给出明确的中文错误提示与解决建议。
3. **新手引导**：对首次使用的教师提供简短的引导流程，说明试卷上传、作业发布等核心功能的操作路径。
4. **适配性**：界面采用响应式设计，在 PC 端与平板端均能正常显示与操作，适配不同屏幕尺寸。

### 3.3.4 可扩展性需求

为应对未来教学需求的变化与技术的迭代，系统需具备良好的可扩展性：

1. **功能扩展**：系统架构采用模块化设计，各功能模块之间低耦合。预留接口扩展点，支持未来新增学科分类、题型类型（如连线题、排序题）或功能模块（如在线考试、口语评测）。
2. **技术扩展**：大模型调用层采用工厂模式设计，支持切换不同的大模型供应商（如 OpenAI、DeepSeek、文心一言等）而无需修改核心业务逻辑。文档解析引擎也支持类似的热插拔替换。
3. **数据扩展**：数据库设计考虑数据量增长场景，建立合理的索引策略，支持百万级试题数据的高效查询。系统应具备水平扩展能力，当用户规模增长时可增加服务器节点分担负载。

### 3.3.5 可靠性需求

作为教学辅助工具，系统的可靠性直接影响正常的教学活动：

1. **系统稳定性**：系统在正常部署环境下支持 7×24 小时持续运行，无宕机、闪退等严重稳定性问题。关键业务路径（试卷解析、作业提交）具备异常捕获与自动恢复机制。
2. **数据可靠性**：数据库配置定时备份策略，每日自动备份核心业务数据。关键操作（如作业发布、成绩修改）记录操作日志，便于问题追溯。
3. **容错性**：当大模型接口调用超时或返回格式异常时，系统能够进行自动重试（最多 3 次），并在最终失败时返回明确错误提示，不导致整个解析流程崩溃。文件上传过程中如遇网络中断，支持断点续传或提示用户重新上传。

## 3.4 系统可行性分析

### 3.4.1 技术可行性

从技术角度分析，本系统所选技术栈均为经过广泛验证的成熟方案：

**前端技术**：Vue 3 作为主流前端框架之一，拥有活跃的社区生态与完善的中文文档。Composition API 提供了更好的逻辑复用与代码组织方式。Vite 构建工具在开发体验与构建速度上均优于传统的 Webpack，适合中小型项目的快速迭代开发。KaTeX 作为轻量级公式渲染引擎，渲染速度约为 MathJax 的 10 倍，能够满足试题展示中大量公式的即时渲染需求。

**后端技术**：Node.js 基于事件驱动与非阻塞 I/O 模型，在处理文件上传、并发请求等 I/O 密集型场景时具有天然优势。Express 框架简洁成熟，社区中间件资源丰富，能够满足系统路由管理、请求处理、错误拦截等核心需求。

**数据存储**：PostgreSQL 是功能全面的关系型数据库，支持 JSON/JSONB 数据类型，既能保证结构化数据的关系完整性，又为试题等非结构化数据提供了灵活的存储方案。

**文档解析与 AI**：MinerU 是开源文档版面分析工具，在中文文档处理上具有良好的表现。大语言模型通过 OpenAI 兼容的 API 接口进行调用，接入方式标准化，可根据实际需求替换不同的模型供应商。

综上所述，系统在技术层面完全可行，开发难度适中，不存在不可攻克的技术障碍。

### 3.4.2 经济可行性

从经济角度分析，本系统的开发与运营成本可控，投入产出比较高：

**开发成本**：系统使用 Vue 3、Node.js、PostgreSQL、MinerU 等均为开源免费技术，无需购买商业软件授权。开发环境仅需普通 PC 与 VS Code 等免费工具，初期硬件投入几乎为零。

**部署成本**：系统为轻量级 Web 应用，初期部署可选择一台 4 核 8G 的云服务器即可满足 200 人以内的使用需求，月度服务器成本约 100-200 元。数据库与后端服务可部署在同一台服务器上，进一步降低运营成本。

**人力成本节省**：以一名数学教师为例，传统方式每周约需 3-4 小时处理试卷录入与作业批改，使用本系统后可缩短至 30 分钟以内。按一个学校 50 名教师计算，每周可节省约 150 小时的工作时间，经济效益显著。

**长期维护成本**：系统采用模块化架构，功能迭代与 bug 修复的维护成本可控。大模型 API 调用按量计费，单份试卷的解析成本约为 0.1-0.3 元，在可接受范围之内。

### 3.4.3 操作可行性

从用户操作角度分析，系统设计充分考虑了中小学师生的技术水平与使用习惯：

**教师群体**：系统界面采用极简设计原则，核心功能入口一目了然。上传试卷、发布作业、查看学情等主要操作均不超过 3 步即可完成。教师无需具备编程或计算机专业知识，经过 10-15 分钟的简单体验即可熟练使用。

# 4 系统总体设计

## 4.1 设计原则

1. 用户中心原则：以教师、学生需求为核心，优化界面与操作流程。

## 4.2 系统总体架构设计

### 4.2.1 表现层

### 4.2.2 业务逻辑层

### 4.2.3 引擎服务层

引擎服务层提供文档解析、大模型调用、公式渲染等核心能力支持。

### 4.2.4 数据持久层

数据持久层负责数据的存储与读取，基于 PostgreSQL 实现，保障数据安全与查询高效。

### 4.2.5 系统交互流程

用户操作→前端界面→后端接口→业务逻辑处理→引擎服务调用→数据库操作→返回结果→前端展示，全流程异步处理，提升响应速度。

## 4.4 智能文档解析核心流程设计

智能文档解析是平台核心功能，流程分为 8 个步骤，全流程自动化完成：

1. 文件上传：教师上传 DOCX/PDF 试卷，系统校验文件格式与大小。
2. 文档预处理：后端接收文件，存储至服务器，生成文件唯一标识。
3. MinerU 解析：调用 MinerU 引擎，完成图文分离、公式识别、版式还原，输出 Markdown 文本。
4. 图片映射：将解析后的图片替换为服务器静态资源链接，实现图文对位。
5. 文本切片：通过异步并发切片算法，将长文档按试题拆分，提升解析效率。
6. 大模型抽取：调用大模型，按指定 JSON 格式抽取试题结构化数据。
7. 高容错解析：通过自研 JSON 容错算法，修正大模型输出格式错误，避免崩溃。
8. 数据入库：清洗试题数据，存入 PostgreSQL 数据库，完成解析。

### 4.4.1 核心算法设计

1. 异步并发切片算法：将长文档分割为多个片段，并发调用大模型，解析效率提升 3 倍。
2. 高容错 JSON 解析算法：针对 LaTeX 公式反斜杠转义、JSON 格式缺失问题，通过正则替换、暴力转义、字段补全，实现 99.9% 的解析成功率。

2. 后端配置：配置数据库连接、JWT 密钥、文件上传路径、MinerU 调用地址、大模型 API 地址。
3. 引擎配置：部署 MinerU 服务，配置模型参数，保证解析准确率。

## 5.2 前端模块详细实现

### 5.2.1 项目初始化

使用 Vite 创建 Vue 3 项目，安装依赖：Vue 3、Tailwind CSS、Axios、KaTeX 等。

### 5.2.2 路由配置

配置前端路由，基于 Vue Router 实现权限拦截。路由守卫通过检查 `localStorage` 中的 JWT Token 判断登录状态，根据用户角色（student/teacher）控制页面访问权限。

### 5.2.3 文件上传组件

使用原生 HTML 文件输入组件结合 Vue 的 `ref` 引用实现文件上传。教师点击上传区域触发隐藏的 `<input type="file">`，选择 DOCX/PDF 文件后通过 change 事件读取文件并发送至后端 API。上传区域支持图片和 PDF 的即时预览。

### 5.2.4 公式渲染实现

封装 KaTeX 渲染函数，在试题展示区域调用，支持行内公式与块级公式渲染，确保分式、积分、化学方程式等复杂公式清晰显示。

### 5.2.5 沉浸式答题组件

采用 Vue 3 组件化开发，答题界面包含题干展示、选项渲染、答案提交等模块。每题作答后自动保存进度，支持客观题即时批改与成绩反馈。

### 5.2.6 AI 答疑界面

实现输入框与 AI 回答展示区域，支持流式输出效果，提供解题思路指导，不直接给出答案。

## 5.3 后端模块详细实现

### 5.3.1 数据库连接配置

系统使用 PostgreSQL + UUID 主键，通过 Node.js `pg` 库连接数据库，配置连接池参数。

### 5.3.2 用户登录接口

用户登录接口验证邮箱与密码，密码使用 bcrypt 进行哈希比对，验证通过后签发 JWT Token 返回前端。

### 5.3.3 班级管理接口

提供班级创建、成员添加、成员列表查询等接口，教师可通过班级邀请码（class_invites 表）让学生加入班级。

### 5.3.4 作业发布接口

教师选择班级和题目（question_ids），设置截止时间与作业状态（草稿/发布），系统将作业写入 assignments 表并推送通知至对应班级学生。

### 5.3.5 作业提交接口

学生提交作答内容，客观题自动批改计分，主观题标记待批改状态，教师可在批改界面手动评分。

### 5.3.6 错题本接口

学生答错试题后自动录入 error_book 表，记录题目 ID、学生错误答案（student_answer）等信息。支持错题列表查询、分类筛选与重做功能。

# 7 总结与展望

## 7.1 工作总结

本文围绕智能教辅平台的设计与实现展开全面研究，针对传统教辅平台试卷处理效率低、图文公式处理难、缺乏智能交互等痛点，整合大语言模型、MinerU 文档版面分析、前后端分离开发技术，完成了以下工作：

1. 完成需求调研与分析，明确中小学师生核心需求，确定系统功能与性能指标。
2. 研究前端 Vue 3、后端 Node.js、MinerU、大模型等核心技术，搭建技术框架。
3. 设计系统四层架构、六大功能模块、智能解析核心流程与数据库结构。
4. 实现前端界面、后端接口、智能解析引擎、核心容错算法的开发，攻克长文档解析、公式转义崩溃、大模型输出错乱等技术难题。
5. 对系统进行功能、性能、兼容性测试，优化系统缺陷，验证系统可用性。

本平台实现了试卷一键解析、题库自动化管理、作业全流程处理、智能错题本、AI 助教答疑的一体化服务，大幅降低教师教学负担，提升学生学习效率，符合教育数字化发展趋势。

## 7.2 研究不足

在研究与开发过程中，仍存在一些不足：

1. 文档解析方面：对极特殊版式、手写字体的试卷解析准确率有待提升。
2. 大模型方面：依赖第三方 API，本地部署优化不足，响应速度可进一步提升。
3. 功能方面：暂未支持主观题自动批改、多学科资源整合。
4. 界面方面：个性化主题、交互动效可进一步优化，提升用户体验。

## 7.3 未来展望

针对研究不足，未来将从以下方面优化升级：

1. 技术优化：微调垂类大模型，提升试卷解析准确率；部署轻量化本地模型，实现私有化部署，降低 API 依赖。
2. 功能扩展：新增主观题 AI 批改、语音答疑等功能，覆盖全场景教辅需求。
3. 界面升级：增加个性化主题、暗黑模式，优化移动端体验。
4. 数据挖掘：基于学情数据实现个性化试题推送、学习路径规划，打造真正的个性化智能教辅平台。
5. 场景拓展：将平台推广至更多教育场景，适配不同年龄段的教育需求。
